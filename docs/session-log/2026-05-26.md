### Crash fixes (commit `a71863b`)

Three stacking bugs found in the M5 animation code shipped 2026-05-25:

1. **Infinite re-render loop in
   [src/hooks/useHistory.ts](../../src/hooks/useHistory.ts).**
   `useHistory` returned a fresh object literal every render. An effect
   in [src/App.tsx](../../src/App.tsx) (`[anim.ready, anim.currentIndex,
   history]`) called `history.reset()` whenever those changed, which
   called `bump()`, which re-rendered useHistory, which produced a new
   returned object, which triggered the effect again. The loop ran at
   frame rate, leaking an `ImageBitmap` per iteration. Eventually OOM
   on iPad.
2. **Pointer listener thrashing in
   [src/components/Canvas.tsx](../../src/components/Canvas.tsx).**
   `handleStrokeEnd` closed over the unstable `anim` object, so Canvas
   tore down + re-bound its pointer event listeners on every App
   render. Combined with bug 1 this made mid-stroke listener swaps a
   common occurrence.
3. **Cross-frame undo silently corrupted data.** Undoing on Frame 2
   restored Frame 1's `ImageBitmap` to the canvas while `currentIndex`
   stayed at 2; the next stroke commit would overwrite Frame 2 with
   Frame 1's content. Semantic bug, not just a crash.

Fixes:
- Rewrote [src/hooks/useHistory.ts](../../src/hooks/useHistory.ts) with
  per-frame stacks keyed by `frameId` (`Map<string, FrameStack>`).
  Switching frames swaps stacks. Returned object is `useMemo`-stable.
  New API: `push`, `undo`, `redo`, `ensureBaseline(frameId)`,
  `dropFrame(frameId)`, `clearAll()`.
- Wrapped `useAnimation` return in `useMemo`; exposed `currentFrameId`.
- App.tsx destructures stable callbacks (`commitCurrentFrame`,
  `historyPush`, etc.) and uses them directly in `handleStrokeEnd` —
  Canvas listeners bind once and stay bound.
- Replaced the broken auto-reset effect with
  `ensureBaseline(currentFrameId)` that initializes a frame's history
  baseline once on entry.
- Gated `useAnimation`'s initial load with `loadedRef` to avoid
  StrictMode double-creating the initial blank frame in dev.
- IndexedDB persist debounce raised from 600ms to 1500ms.

### Canvas sizing fix (commit `c8891c9`)

After deploying the crash fixes, Emmett reported the canvas was all
grey, no marks appeared on cursor input, and the onion-skin icon was
missing. The M5 refactor (M0→M5) had positioned all canvas-stack
children as `position: absolute; inset: 0` — which meant nothing in
the wrapper contributed to its size. CSS `aspect-ratio` +
`max-width:100%` + `max-height:100%` with no definite anchor in the
percentage chain caused `.canvas-stack` to collapse to 0×0. The
drawing canvas rendered at 0×0 and OnionControls (at `top: 10px;
right: 10px`) was clipped out of the zero-size container.

Fix: gave up on pure-CSS aspect-ratio sizing through layered absolute
children. Added a `ResizeObserver` in
[src/components/Canvas.tsx](../../src/components/Canvas.tsx) that
watches `.canvas-frame` and imperatively sets explicit pixel
width/height on `.canvas-stack` to the largest 8:5 box that fits.
`useLayoutEffect` runs before paint so there's no flash of zero-size
content. Also removed the redundant `.canvas-bg` div — the
`.canvas-stack` itself has `background: white`.

## Decisions made

- **ResizeObserver beats pure-CSS aspect-ratio for layered canvases.**
  When the layout involves stacked absolute-positioned children, CSS
  aspect-ratio doesn't reliably size a wrapper. Imperative sizing in
  `useLayoutEffect` is deterministic, ~10 lines, and bulletproof
  across browsers. Worth the trade.
- **Per-frame undo is the right model.** Originally listed as
  "follow-up polish" — turned out to be required for correctness
  (cross-frame undo was a data-loss bug, not just bad UX).
- **Memoize hook return objects when they live in effect dependency
  arrays.** Otherwise consumers get reference churn that triggers
  spurious effect runs and can compound into render storms. Applied
  to `useHistory` and `useAnimation`.
